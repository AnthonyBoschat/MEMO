Informations pour la création d'un gestionnaire d'erreur
• Par défaut, c'est le gestionnaire d'erreur PHP qui est utiliser

• La définition d'un gestionnaire d'erreur peut prendre plusieurs paramètres obligatoires et optionnelles
( Les noms des paramètres sont des conventions, rien d'obligatoire )

• $errno -> Type d'erreur

E_USER_NOTICE -> Avertissement
E_USER_WARNING -> Alerte
E_USER_ERROR -> Erreur
• $errmsg -> Le message d'erreur
• $errfile -> Le fichier ou il y a une erreur
• $errline -> La ligne ou il y a une erreur
• $errdatas -> Les variables utiliser lors du contexte
Créé son propre gestionnaire d'erreur
Il est utile d'utiliser un Switch dans ce cas


function mon_gestionnaire_erreur($errno, $errmsg, $errfile, $errline)
{

    switch($errno)
    {
        case E_USER_NOTICE:
            echo $errmsg;
            break;
        
        case E_USER_WARNING:
            echo $errmsg." sur le fichier ".$errfile;
            break;

        case E_USER_ERROR:
            echo $errmsg;
            break;

        default:
            echo "Erreur non reconnue...";
            break;
    }

}

// Ou

$mon_gestionnaire_erreur = function(#paramètres)
{

    // Conditions des erreurs

};

Principe
•Un cookie est un fichier texte enregistrer sur la naviguateur web

• Il stock des informations (pas de sécurité, ni confidentiel )
Créé un cookie
Le cookie doit être créé avant toute requête http, tout autre code

setcookie("#nom_du_cookie", "#valeur_du_cookie", "#temp_avant_expiration", "#chemin_du_cookie", "#nom_de_domaine", #cookie_securiser?", "#acces_uniquement_en_http");

• nom_du_cookie : "string"
• valeur_du_cookie : "string"
• temp_avant_expiration : time() + x (time() + la durée voulu)

• nom_de_domaine : "string" | null (si nom de domain)
• cookie_securiser : true | false (https?)
• acces_uniquement_en_http : true | false (http et pas javascript, presque toujours true)

    2. (Il n'y a pas besoin de lui renseigner les attributs $_name et $_description, dans la classe mère elles sont en protected, donc accessible depuis la classe fille)

Utilisation de fetch
paramètre de fetch :

• BOTH ( par défaut : On récupère les informations de la base de donnée, et les index de ligne )
• ASSOC ( On récupère les informations de la base de donnée, sans les index de ligne )
• OBJ ( retourne un objet )

( L'utilisation de ASSOC ou de OBJ est une question de préférence )

On peut l'utiliser uniquement au moment de l'affichage :
function requete($PDO)
{
    // On lance une requête, 
    // On stock le retour de cette requête dans $result
    $result = $PDO -> query("SELECT * FROM fv_countries");

    // On parcours tout ce que contient $result 
    // On change son mode de retour fetch
    while($data = $result->fetch(PDO::FETCH_ASSOC))
    {
        // On affiche chaques informations
        echo "< pre>";
        print_r($data);
        echo "< /pre>";
    }
}

Ou on peut l'utiliser avant avec une méthode PDO, ce qui affectera toute les manipulation de $result ensuite :
function requete($PDO)
{
    // On lance une requête, 
    // On stock le retour de cette requête dans $result
    $result = $PDO -> query("SELECT * FROM fv_countries");

    // On change son mode de retour fetch
    $result->setFetchMode(PDO::FETCH_ASSOC);

    // On parcours tout ce que contient $result
    foreach($result as $data)
    {
        // On affiche chaques informations
        echo "< pre>";
        print_r($data);
        echo "< /pre>";
    }
}

Faille par injection SQL
C'est le fait d'injecter du code et d'alterer une requête SQL pour récupérer des mails, des mots de passes par exemple.

Pas encore sécuriser
$sql = "SELECT * FROM fv_countries";
Un hacker peut intégrer des simples quotes et rentrer du code :
$sql = "SELECT * ''FROM fv_countries";

On peut sécuriser ça avec une requête préparée
Requêtes préparée
Il faut définir une nouvelle option dans $options :
PDO::ATTR_EMULATE_PREPARES => false // Pas d'emulation de requête préparée

Dans le code :
function requete($PDO)
{
    // On prépare une requête, qui est stocker dans la variable $requete
    $requete = $PDO->prepare("SELECT * FROM fv_countries");
    // On lance la requête quand on en a besoin :
    $requete->execute();


    echo "< pre>";
    // fetchAll pour obtenir un tableau associative
    print_r($requete->fetchAll(PDO::FETCH_ASSOC));
    echo "< pre>";
}

Si la syntaxe de la requête est incorrecte, ça renverra une erreur :
$requete = $PDO->prepare("SELECT * ''FROM fv_countries");

Imaginons que nous voulons maintenant toute les informations lorsque id_country est spécifique :
$requete = $PDO->prepare("SELECT * FROM fv_countries WHERE id_country = ?");
// ou pour nommer l'identificateur
$requete = $PDO->prepare("SELECT * FROM fv_countries WHERE id_country = :id");

? et :id sont des identificateurs, ils signifient que c'est une valeur qu'on definira plus tard.

Pour definir cette valeur avec bindValue :
$requete->bindValue(1, 3);
// 1 / :id = première valeur à définir ( premier identificateur )
// 3 = la valeur à renseigner

L'avantage, c'es qu'on pourra rappeler cette requête avec des valeurs différentes. bindValue évalue la requête au moment de l'utilisation de bindValue


Pour definir cette valeur avec bindParam :
$requete->bindParam(1, $id);

$id = 3;
L'avantage de bindParam, c'est que le paramètre peut être définie après bindParam. Il évalue la requête au moment de l'execution ( execute() )
Verifier le type de donnée à passer
En général, le SGBD converti automatiquement
bindValue(":id", 3, PDO::PARAM_STR) // Par défault, passe un string
bindValue(":id", 3, PDO::PARAM_BOOL) // Passe un booléen
bindValue(":id", 3, PDO::PARAM_INT) // Passe un entier / un float
